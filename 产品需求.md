## 1. 项目背景与目标

### 1.1 痛点描述

当前的桌面端输入法体验基本处于停滞状态，主要依赖 N-gram 概率与静态本地词库，缺乏对长程上下文（Long-range Context）的理解。在用户输入生僻术语、专业缩写或发生拼音模糊时，输入法往往只能在现有候选词中"矮子里拔高个"。此外，传统大模型若采用多步自回归（Auto-regressive）生成，会带来不可接受的输入延迟，导致"AI 赋能"与"打字流畅度"相互互斥。

### 1.2 核心目标

利用 Windows 平台充沛的本地算力，构建一款基于端侧大模型（Edge LLM）的上下文感知输入法。通过**极轻量化部署**、**两阶段推理选词**以及**企业级并发状态同步**，实现零配置开箱即用。在严格保障低延迟的前提下，不仅对现有候选词进行高精度重排，更能突破本地词库天花板，实现贴合上下文的"候选词感知与无中生有"。

## 2. 用户场景定义

- **场景 A (上下文感知选词)：** 用户在前文提到了"神经网络"，输入 `moxing` 时，系统感知上下文，从原生候选词中将"模型"选出并提升至 2 号位，用户按 `2` 即可选择。
- **场景 B (拼音纠错)：** 用户想输入"模型"但误打成 `mingxing`，AI 从候选词列表中选出"模型"而非默认的"明星"，实现基于上下文的智能纠错。
- **场景 C (词库穿透与感知生成)：** 用户正在撰写海洋环境监测相关的学术论文，前文涉及"弱监督语义分割"，输入拼音 `weijiandu`。即便本地四叶草词库中未收录该长词，大模型也能通过上下文推理直接生成"微监督"等关联专业词汇并推至 2 号位，经拼音碰撞校验后上屏。
- **场景 D (连续选词)：** 用户输入 `damoxingxunlian`，AI 推荐"大模型"，选择后剩余拼音 `xunlian` 自动保留，用户可继续选择"训练"等后续候选词，实现流畅的多段输入。
- **场景 E (个性化瞬时内化)：** 用户曾将 AI 推荐词手动纠正，系统静默记录。下次在相似上下文中输入，系统以毫秒级速度查表（SQLite）并直接提权，越用越顺手。

## 3. 功能需求

### 3.1 核心功能：上下文感知引擎

#### 3.1.1 全局上下文抓取 (UIA Context Fetching) ✅ 已完成

通过 Windows UIAutomation (UIA) 接口在后台看门狗线程中持续探测光标前文本，以 500ms 间隔更新全局上下文缓存。

**三级瀑布策略：**

| 优先级 | 策略                | 适用场景                   | 实测耗时           |
| --- | ----------------- | ---------------------- | -------------- |
| 1   | TextPattern 字符级回退 | Word、WPS、记事本等富文本编辑器    | \~5-10ms (优化后) |
| 2   | ValuePattern      | 浏览器地址栏、VS Code 等纯文本输入框 | \~2-6ms        |
| 3   | Name 属性保底         | 非标控件将文字藏在 Name 属性中     | \~1ms          |

**关键优化历史：**

- 初版采用 `ExpandToEnclosingUnit(TextUnit.Paragraph)` 获取段落文本，实测稳定在 \~1001ms，原因是该 API 触发 Word 排版引擎的段落边界计算。
- 优化后改用 `MoveEndpointByUnit(TextUnit.Character, -150)` 直接从光标处向左移动 150 个字符，绕过段落排版，耗时降至 \~1-5ms，提速约 **200 倍**。

**当前配置：**

- 上下文窗口：光标前 150 个字符
- 轮询间隔：500ms
- COM 初始化：每个线程独立 `pythoncom.CoInitialize()`
- 全局搜索超时：`auto.SetGlobalSearchTimeout(0.3)`（300ms）

#### 3.1.2 两阶段推理引擎 (Two-Phase Inference) ✅ 已完成

采用两阶段架构，针对小模型（1.5B）的能力量身设计：

**Phase 1 — 选择策略 (Selection)：**

- 将原生候选词的前 5 个作为"选择题"选项提交给大模型
- 模型根据上下文和拼音，从候选词列表中选出最合适的词
- 若选出的词与原生首位不同，则作为 AI 推荐词注入 2 号位
- 对小模型而言，"选择题"远比"问答题"可靠

**Phase 2 — 生成策略 (Generation)：**

- 仅在 Phase 1 未命中且拼音长度 ≥ 3 时触发
- 模型从上下文和拼音直接生成一个中文词
- 生成结果必须通过拼音碰撞验证（`pinyin_is_close`，阈值 60%）
- 防止模型"胡说"，确保生成的词与用户输入拼音基本吻合

**推理配置：**

- 模型：`Qwen2.5-1.5B-Instruct` Q4\_K\_M 量化版
- 推理框架：`llama-cpp-python`，全量 GPU 转移（`n_gpu_layers=-1`）
- 上下文窗口：32768 tokens（`n_ctx=1024*32`）
- 采样策略：`temperature=0.0`（贪心解码，确定性输出）
- 输出约束：`max_tokens=6`，`stop=["\n", "。", "，", "、"]`
- 上下文截取：光标前 100 字符（`clean_context[-100:]`）

#### 3.1.3 精确候选词边界控制 (Precise Candidate Boundary) ✅ 已完成

AI 生成的候选词携带 `covered_length` 字段（生成词的拼音长度），Lua 端据此精确设置候选词的 `start`/`end` 边界：

```
示例：输入 "damoxingxunlian"
- AI 生成 "大模型"，covered_length = 8 (da+mo+xing)
- Lua 设置 safe_start=0, safe_end=8
- 用户选择 "大模型" 后，仅消耗 "damoxing"，剩余 "xunlian" 可继续选择
```

#### 3.1.4 候选词重排与渲染顺序 ✅ 已完成

**候选词重排 (Rerank)：**

基于模型 logprobs 对原生候选词进行真实打分重排。单次前向传播获取 top-20 token 概率，将候选词首字映射到对应的 logprob 作为置信度分数，按分数降序排列返回 `reranked_candidates`。

```
Python 端返回示例：
{
  "generated_word": "模型",
  "covered_length": 6,
  "reranked_candidates": [
    {"word": "模型", "score": -0.3215},
    {"word": "明星", "score": -1.8742},
    {"word": "模样", "score": -2.1567},
    ...
  ]
}
```

**Lua 端渲染顺序：**

```
1号位：四叶草原生首位（保底，用户闭眼敲空格体验不变）
2号位：Phase2 生成词（仅当不在原生候选中时注入，带 🌟 标记）
重排区域：按 reranked_candidates 顺序输出（Phase1 推荐词自然靠前）
补漏：未被重排覆盖的原生候选词
后续：四叶草词库全量继续释放（翻页键完美工作）
```

### 3.2 通信与可靠性保障

#### 3.2.1 TCP Socket 通信 ✅ 已完成

- Python 后端：`127.0.0.1:5000`，常驻本地 TCP 监听服务
- Lua 前端：通过 `package.cpath` 动态寻路加载 64 位 `luasocket`，绕过 RIME 沙盒
- 协议格式：HTTP/1.1 简化版（Lua 发送 HTTP POST，Python 返回 HTTP 200）
- JSON 解析：Lua 端纯正则提取关键字段（`generated_word`、`covered_length`、`reranked_candidates`），无外部 JSON 库依赖
- 请求类型：`rerank`（AI 推理）和 `correction`（纠错记录），通过 `type` 字段分发

#### 3.2.2 微服务级故障熔断 (Circuit Breaker) ✅ 已完成

- Lua 侧引入全局监控，TCP 连接失败一次，立即将后端标记为"死亡"
- 进入 5 秒冷却期（`CIRCUIT_BREAKER_COOLDOWN = 5`）
- 冷却期内所有按键请求执行 0ms 物理放行，杜绝后台服务异常导致的输入法卡顿

#### 3.2.3 CAS 版本号防脏读 ✅ 已完成

- 每次 Lua 发起请求携带唯一 `version_id`（`os.clock() + "_" + current_pinyin`）
- Python 返回结果时携带相同 `version_id`
- 若用户已追加打字（版本号变更），废弃返回包被丢弃，杜绝 UI 闪烁

#### 3.2.4 触发规则 ✅ 已完成

- 仅当用户输入拼音长度 > 5 个字符时才发起 TCP 通信
- 高频连击放任，通过后端 CAS 机制化解压力
- TCP 超时：200ms（`tcp:settimeout(0.2)`）

### 3.3 基础输入功能 (基于 RIME) ✅ 已完成

- **原生 Candidate 安全渲染：** 废弃易引发 C++ 内存越界的 `ShadowCandidate`，采用最底层的原生 `Candidate` 结合属性解构来重新构造带 AI 标记的候选词，确保 RIME 引擎坚如磐石。
- **白名单安全放行：** 遇到四叶草中的颜文字、特殊标点或组合指令时，Lua 拦截器无条件直接放行，确保系统不崩溃。
- **Lazy 加载候选词：** 仅截取前 5 个候选词发送给后端，绝不遍历几千个词拖垮性能。

### 4.4 SQLite 记忆体系 ✅ 已完成

**数据库结构 (`data/memory.db`)：**

```sql
CREATE TABLE corrections (
    id INTEGER PRIMARY KEY,
    context_hash TEXT NOT NULL,    -- 上下文 MD5 hash (取后50字符)
    pinyin TEXT NOT NULL,          -- 用户输入的拼音
    correct_word TEXT NOT NULL,    -- 用户实际选择的词
    created_at REAL NOT NULL,      -- 创建时间戳
    last_used REAL NOT NULL,       -- 最后使用时间戳
    use_count INTEGER DEFAULT 1,   -- 使用次数
    UNIQUE(context_hash, pinyin, correct_word)
)
```

**工作流程：**

1. **L1 Cache 查询**：推理前先查询 SQLite，若 (context\_hash, pinyin) 命中，直接提权对应词，跳过模型推理
2. **纠错捕获**：Lua 端通过 `commit_notifier` 检测用户选词，若上屏文本不含 AI 推荐词，发送 `type=correction` 请求；Python 端使用 UIA 上下文作为 context 异步写入 SQLite
3. **LRU 淘汰**：定期清理超过 30 天未使用或超量（>10000条）的记录

**配置参数 (`config/config.yaml`)：**

```yaml
memory:
  db_path: "data/memory.db"
  max_records: 10000
  max_age_days: 30
  cleanup_interval: 3600
  context_hash_len: 50
```

**纠错接口：**

- Lua 端发送 `type: "correction"` 请求，携带 pinyin、correct\_word
- Python 端自动使用当前 UIA 上下文作为 context，异步写入 SQLite
- 写入采用 UPSERT 语义，重复记录自动累加 use\_count

## 5. 技术方案概要

### 5.1 解耦架构设计

1. **前端 (Lua 同步拦截器)：** 依托 `package.cpath` 动态寻路注入，绕过 RIME 沙盒完美加载 64 位 `luasocket`。负责拼音防抖、200ms TCP 超时、5s 故障熔断、CAS 版本号生成、正则无依赖解析后端 JSON，通过原生 `Candidate` + `covered_length` 精确边界重构并输出 AI 候选词，以及通过 `commit_notifier` 捕获用户选词并发送纠错请求。
2. **后端 (Python 极速引擎)：** 常驻本地的 TCP 监听服务。承载 UIA 后台看门狗上下文抓取、两阶段推理引擎（选择 + 生成）、`pypinyin` 拼音碰撞验证、`covered_length` 计算、logprobs 候选词重排、SQLite 记忆体系（L1 Cache + 纠错捕获 + LRU 淘汰），以及 `llama-cpp-python` 大模型推理。

### 5.2 系统工作流 (Workflow Pipeline)

系统一次完整的被动触发打字生命周期如下：

1. **输入拦截层 (Lua)：** 用户输入拼音长度 > 5，触发 Lua Filter。检查熔断器是否冷却中，若后端存活，生成 `Version_ID`（`os.clock_pinyin`）。
2. **网络发射 (Lua → Python)：** Lua 经由 TCP Socket 发送 HTTP POST（JSON 报文，阻塞等待，超时上限 200ms）。
3. **AI 推理层 (Python)：**
   - 剥离 HTTP 头部，提取 JSON Body。
   - 从全局上下文缓存读取当前光标前文本（由后台看门狗线程持续更新）。
   - 清洗上下文（去除末尾英文、截取最近 100 字符）。
   - **L1 Cache 查询：** 先查询 SQLite 记忆库，若 (context\_hash, pinyin) 命中，直接提权对应词，跳过模型推理。
   - **Phase 1 (选择)：** 将候选词前 5 个提交给大模型，选出最符合语境的词。若选出的词不同于原生首位，记为 `generated_word`。
   - **Phase 2 (生成)：** 若 Phase 1 未命中，将拼音和上下文提交给大模型直接生成候选词，并通过拼音碰撞验证（60% 相似度阈值）。
   - **候选词重排：** 使用 logprobs 获取 top-20 token 概率，对候选词进行真实打分排序。
   - 计算 `covered_length`（生成词的拼音长度），用于精确控制候选词边界。
4. **闭环渲染层 (Lua)：** Python 发回 HTTP 200 JSON 响应。Lua 纯正则解析，判断 `Version_ID` 是否一致。若一致：
   - 1 号位：四叶草原生首位（保底）
   - 2 号位：Phase2 生成词（仅当不在原生候选中时注入，带 🌟 标记）
   - 重排区域：按 `reranked_candidates` 顺序输出
   - 补漏 + 词库全量续放
   - 若版本不一致或超时，放行原生词。
5. **纠错捕获 (Lua → Python)：** 用户选词上屏时，`commit_notifier` 检测上屏文本是否包含 AI 推荐词。若不包含，发送 `type=correction` 请求，Python 异步写入 SQLite。
